Android Room

RoomDatabase

一个单例类，供应用全局使用

ObjectDao

添加注解后成为 有关 CURD 的方法描述

Object

添加注解后成为 对数据库中一行数据的映射

LiveData

CURD 操作都需要更新视图，本质上是因为数据变更

DRY：异步监听数据变更，使得更新视图的方法只需写出一次

AsyncTask

数据库操作被假定为非常耗时，因此不建议在主线程中使用，对于每个 CURD 操作生成一个单例类，用以隐藏其余线程

ObjectViewModel

ViewModel 负责管理界面的数据，精简 Activity

ObjectRepository

ViewModel 不应看到增删查改的具体实现 ———— ObjectDao RoomDatabase AsyncTask 等，这些类被隐藏在 ObjectRepository 后

Repository 具体以何种方式取得数据（网络，本地数据库，随机生成，等），ViewModel 不可见

总结

数据库相关

  • Database
  • Object
  • ObjectDao

界面相关

  • LiveData
  • Activity
  • ObjectViewModel

读书笔记 - MongoDB实战(第二版)

这本书快过借阅期限了，不打算再续借，所以暂时不写作业了，把这书再读一读。

NoSQL 的本意是 Not Only SQL ，不过我看这本书的时候感受到的最大的不同确实是 No SQL ，众所周知数据库里存的东西拿到程序里来总是会变成一个一个对象，所以即便数据库本身是不是这么回事，在不会直接接触的程序看来它还是能看成一个对象的集合，集合约等于一下就是数组，平时用的这些编程语言里都能写出来对象的数组，所以得出结论 —— 不一定非得用 SQL 语句去操作集合，用别的东西也行。

个人理解

MongoDB 里面，“一张表”（和关系型数据库里同种地位的那种东西，这儿叫集合）就是一个表对象，用 Json 对象来描述一个对数据库的操作，把 Json 对象传给表对象的某个方法（find update 之类的），就会执行这个动作。当然这个代表操作的 Json 对象的结构可不是瞎写的，所以还是有一门查询语言要学，但是比单独学一门 SQL 要强 —— 可能是由于找不到专门的手册类的教材，之前光是对几张表取连接就难住了我很久，更不用说除法了。而这个语言真想学，去看 MongoDB 的官方文档就行

回到这本书上来，前几章一直在讲查询语言，逐步的介绍各种关键字，接着介绍了聚合框架（aggregation framework），可以理解为链式编程，在 JS 里用过 Jquery 或者 Promise 肯定能明白我在说什么。因为不可能把所有数据取到在跑的程序里来再做筛选和计算，所以这些东西只好在数据库里就被描述为几个操作提前做完（也许这确实是数据库该做的事情？），

然后就是占全书一半篇幅的第三部分了，名字是 精通 MongoDB ，听起来就像那种存了大量数据的大企业里的某个特殊岗位才会需要了解的东西，细看一下目录，查询优化，分布式，灾备，监控全整上了，我就不看了，至少这次用不着看。

感受

算是一本教程和工具书的结合。这本书某些地方的翻译水平像是机翻，大部分像是我翻，剩下的部分像是没翻，可能这个译者没有程序员的从业经验，术语知道的比较少。搜了一下，发现早已有人吐槽过翻译

ok，总结完了，明天还回去

使用 PowerShell 开关触摸屏

我这个平板到货有一段时间了，由于电池容量在出厂容量的 80% 出头，所以本着能省的电尽量省一省的想法，我把主意打到了触摸屏上面，立在桌面上用键盘的时候就从设备管理器里把它关掉，要换成平板的时候再重新开启。虽然不知道是不是真的能省电，但是心理安慰总是有的。

这样持续了大约有一个星期，我受不了了，今天一直在找有什么办法能自动化，结果确实有，一个很短的 PowerShell 脚本就能搞定：

  • 这个脚本文件的编码需要为 「UTF-8 with BOM」，不然运行会乱码 – @yuuza
  • 把 PowerShell 的运行代码页改成 UTF-8 以后，触摸屏的名字会变成英文 ‘HID-compliant touch screen’，这样反倒用不着特意规定脚本文件的编码了（因为全是 ascii）

阅读的历史记录

这学期有门必修课《软件工程》，刚上完第一节课。按老师的意思，最后要用在课程中学到的方法做点东西出来，我确实想做点东西出来记录每本读过的书。目前这种东西只存在于我电脑桌面上的一个 execl 表格里，很久没有动过了。

我打算用 Java 写一个 Android app 出来，满足这个记录需求

Pros：

  • 应用的布局和行为是分离的，和我已经掌握的 html - js 类似
  • 自带 sqlite 数据库，不用去操心记录具体怎么存
  • 移动端可以简化一些操作，比如添加自己读过的书，就可以调摄像头 OCR 索书号，不用自己手动输入了（这个耗时真的长）
  • 阅读记录的数据可以用在别处，扩展应用的功能，比如阅读时间可视化、发现同样 在读/读过 这本书的同学

Cons：

  • 移动端不适合打很多字，所以会限制写作
  • 我对这个应用需要的工程量大小完全没有概念
  • 严格来讲，我对 Android app 的了解仅限于 helloworld

dotnetmvc

这门课差点挂了，看着 vs 里面一堆解决方案占空，如果放网盘里造福后人又得挨个删依赖拷源码，太麻烦就算了，总结一下这门课教什么就删掉

通用流程

大致就简单介绍了一下后端的东西，一个网络请求到了服务器会不可避免地先经过路由组件，解析 url 并分配到对应的处理程序，在 .net mvc 里处理程序指的就是 controller 类的各种方法，这些方法可能会和数据库交互，这部分按理说也是放哪种服务器都一样，数据库连接，数据库上下文，数据库游标……数据库披萨[doge]

依赖管理

老师直接给了两个 dll，让添加到引用里。用来和数据库互动的，我用了会出版本不符的问题，好在有位朋友帮忙，教我用 nuget 包管理器（这段话听起来很像广告，可能是因为提到了具体的包管理器的名字，不过应该也没有其它的吧）。装最新版的 EntityFramework 就能用了。

安全验证

要手动改 web.config ，加上允许表单验证。这段正好实验课没听到，花了我两天没搞明白，后来又去问老师，然后又查的文档才改对地方，那个 cookie 到底怎么 把/和 这两部分语句联系起来的还是搞不清楚，有点不高兴

个人意见

controller 的注解很有趣，我以前仅仅是听说过这东西，实际用起来相当舒服，就是给参数的方式不习惯，为啥不能做成和函数调用一样呢，省点脑细胞。

无参数的注解，确实给人一种介于注释和代码间的感觉。

我挺讨厌模板的，主要是觉得“不干净”，之前看到 jsp 的时候就是这样,和一门新的语言没啥区别，还是前端 js ajax、fetch 拉数据好，废不了多少时间。hexo 也是模板，好在没有特殊需求的时候我不需要打开井盖去看那部分东西

而且我不知道同时传两个强类型对象给 view 的方法，搞得遇到这种情况的时候得放一个在viewbag 里，用的时候强制类型转换（声明类型），所以更讨厌了

模型？不用 sql 而用 c# 描述数据模型，有点非关系型数据库的建模的感觉，话说 curd 也不用写 sql ，挺爽的，不知道实现具体做了什么。

总结

懵懵懂懂的，结课了也没学懂多少，没兴趣了，感觉 mvc 这种模式并不好，三者之间互动的语言不是同一种，而且模型就真的是模型了，自己没有方法，一点面向对象的思路都没有，还有各种我想要列出但想不出的缺点，反正这东西不好使，我不想继续学就对了。

写完了，这就删库跑路。
